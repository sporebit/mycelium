# Workflow rules

## At the end of every task that changes code

If the task addresses multiple distinct concerns (separate bugs, separate features), commit them separately — one commit per concern. This keeps history clean and lets any individual change be reverted without losing the others.

For each commit:
1. Stage the files relevant to that concern: `git add <files>`.
2. Commit with a short, descriptive message summarising that concern alone (one line, present tense, no "Generated by Claude" footer).

After all concerns are committed, push once at the end: `git push origin main`.

For single-concern tasks, one commit is fine. Use judgment — don't manufacture splits that aren't natural.

Skip this entire flow only for exploratory tasks (reading files, answering questions) with no code changes.

## Before every push

Run `npx next build` and wait for it to succeed. If it fails, fix the underlying issue before pushing — don't push and hope, and don't disable the failing rule.

`tsc --noEmit` is **NOT** sufficient. The production build gates on ESLint (`prefer-const`, `no-unused-vars`, `react-hooks/exhaustive-deps`, etc.) in addition to types. A green `tsc` with a red `next build` is a real and recurring failure mode — the Vercel build for commit `230d686` failed exactly this way (single `prefer-const` error in a new dashboard card that `tsc` accepted).

Do not set `eslint.ignoreDuringBuilds` in `next.config.*` to sidestep this. The gate is the point.

## Database migrations

This project uses the Supabase CLI. Migration files live in `supabase/migrations/`, numbered `NNNN_description.sql`.

To apply a new migration:
1. Write the file at `supabase/migrations/NNNN_description.sql` using the next number in sequence.
2. Run `supabase db push` from the project root. Confirm with `Y` when prompted.
3. Verify with `supabase migration list` — both Local and Remote columns should show the new number.

Never instruct the user to "apply the migration in the SQL editor" — the CLI handles it.

For read-only diagnostic queries (e.g. `SELECT conname FROM pg_constraint...`), use the CLI's exec capability or recommend running in the SQL editor. Never run destructive SQL outside a migration file.
